# Glossary: What Every Keyword Means

## Shield

CARVES Shield is a self-check label for AI code governance. It reads summary evidence and produces Standard G/H/A results, a Lite score, and a local badge.

## Self-check

Self-check means the evidence is generated by the project locally and evaluated by Shield v0 rules. It is useful evidence, but it is not third-party certification.

## Local self-check

Local self-check is the public posture for Shield, Matrix, badges, challenge runs, and verification output in this release. It means the tools evaluate local summary evidence and local artifacts. It does not mean hosted verification, public certification, model safety benchmarking, AI model ranking, source-code security review, semantic correctness proof, or operating-system sandboxing.

## Verified evaluation

Verified evaluation is reserved future language. Shield v0 does not emit verified status, does not certify a project, and does not claim central review.

## Verification result

Verification result means a local verifier checked an existing artifact bundle against its manifest, required files, hashes, sizes, schema metadata, privacy flags, and summary consistency. A successful verification result says the local bundle matches the current verifier contract. It is not Shield verified evaluation, hosted review, certification, a public leaderboard result, a model safety rating, or proof that the source code is correct.

## Evidence

Evidence is a summary document. It records governance facts such as policy presence, CI wiring, decision counts, handoff freshness, and audit readability.

Evidence does not include source code, raw diffs, prompts, model responses, secrets, or credentials by default.

## shield-evidence.v0

The JSON evidence schema supported by the current local evaluator.

## Privacy posture

Privacy posture describes whether the evidence stays inside the expected privacy boundary. The default is summary-first and must not include sensitive material.

## Source upload

Source upload means sending source file content. Shield v0 does not require source upload for self-checks.

## Raw diff

Raw diff means the raw text output of git diff. Shield v0 does not require raw diffs by default. Evidence should use counts, path summaries, or decision summaries instead.

## Guard

Guard is the output governance dimension, shown as `G`. It checks whether AI patches pass declared boundaries such as protected paths, change budgets, fail-closed CI, and decision records.

## Handoff

Handoff is the input governance dimension, shown as `H`. It checks whether the next AI session receives reliable context such as current objective, remaining work, completed facts, must-not-repeat notes, and target repository match.

## Audit

Audit is the history governance dimension, shown as `A`. It checks whether prior AI-related decisions can be read, explained, and reviewed through logs, rule ids, evidence refs, and explain coverage.

## G/H/A

`G/H/A` is the Shield Standard dimension label:

- `G` is Guard evidence for output governance.
- `H` is Handoff evidence for input governance.
- `A` is Audit evidence for history governance.

The three dimensions describe governance maturity. They are not a model safety score, certification level, security audit result, semantic correctness proof, or guarantee that a patch should merge.

## Standard

Standard is the three-dimensional result. It is not averaged into one score. Example:

```text
CARVES G8.H5.A3 /30d PASS
```

Standard keeps weak dimensions visible. A project may be strong in G and weak in H or A.

## Lite

Lite is a quick 0-100 self-check score. It is useful for sharing, comparison, onboarding, and rough posture checks, but it does not replace Standard.

## Lite score

Lite score is the numeric 0-100 projection derived from Shield evidence and the Standard G/H/A posture. It is a convenience readout for local workflow governance maturity. It must not hide weak G/H/A dimensions, and it must not be presented as model safety, certification, hosted verification, public ranking, or source-code correctness.

## PASS

`PASS` means the current Shield Standard evaluation did not hit a critical governance gate for the supplied summary evidence. It is a local self-check result, not a merge approval, certification, hosted verification, public safety label, security audit, or proof that generated code is correct.

## REVIEW

`REVIEW` means a Guard decision or related governance output found risk that needs human inspection before normal merge or release handling. It is a governance workflow signal. It does not mean the model is unsafe, the project is certified unsafe, or the source code is semantically wrong.

## BLOCK

`BLOCK` means a Guard decision or related governance output found a policy violation that should not proceed as-is. Fix, shrink, or explicitly handle the patch before merging. It is a local workflow boundary, not a model safety verdict, certification result, hosted enforcement action, or operating-system sandbox.

## Level 0

`0` means the dimension is not enabled or has no usable evidence. It is not a failure claim.

## Critical Gate

Critical Gate means a small set of mechanical trust failures, such as invalid policy schema, disabled fail-closed posture, or CI that does not block a block result.

## C

`C` means Critical Gate failure. It is not numeric and must not be averaged. `C` is worse than `0` because `0` makes no claim, while `C` means a governance claim exists but failed a critical trust rule.

## Sample window

Sample window is the evidence period, such as `/30d` for the last 30 days. Strong levels such as 8-9 usually require sustained window evidence.

## Badge

Badge is a local SVG label. It can be placed in a README to show self-check output, but it is not a certification mark.

## Challenge result

Challenge result means `carves-shield challenge` evaluated a local Shield Lite challenge pack and reported case pass/fail status, pass rate, and the fixed label `local challenge result, not certified safe`. Challenge results exercise known local governance fixtures. They are not certification, hosted verification, public leaderboard entries, model safety benchmarks, semantic source-code correctness proof, or operating-system sandbox proof.

## GitHub Actions proof

GitHub Actions proof runs local evaluate and badge commands in CI and uploads JSON/SVG artifacts. It does not require provider secrets or a hosted API.
